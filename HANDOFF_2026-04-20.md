## ROOT CAUSE OF HISTORIC BREAK
On 4/19/26, background threading was added to PROCESS RANGE to allow a non-blocking UI.
Resolve's embedded Python sets sys.stdout=None for daemon threads. The first print() call
inside the thread crashed silently, killing the thread before a single frame was processed.
This was NOT obvious — no error, no log, just silence.

## THE FIX (commit 91f1392)
PROCESS RANGE runs synchronously on the main thread — no threading.Thread.
The UI freezes during processing. That is expected and correct.
Status label updates every 10 frames so user can see progress.

## DO NOT ADD THREADING BACK TO PROCESS RANGE
Unless you solve sys.stdout=None in Resolve's daemon threads first.
Patch required: sys.stdout = io.StringIO() before any thread starts.
Even with the patch, cv2.VideoCapture may hang in threads due to COM apartment issues.
The synchronous approach is reliable. Leave it alone.
