## WHAT WAS BUILT THIS SESSION

### 1. SAM2 Video Propagation for BRAW — FIXED
**Problem:** The BRAW synchronous path (lines ~1829–1932) only ever applied a static SAM2 gate — one mask stamped on all frames. `run_sam2_video_propagation()` was only wired into the background thread path (`_run()`), which BRAW skips entirely via an early `return`.

**Fix:** In the BRAW sync block, before the per-frame loop, now calls `run_sam2_video_propagation(braw_frames_dir, 0, 0, 0, dur, ...)` to get per-frame tracking masks. Falls back to static gate if propagation returns nothing.

**What can go wrong:**
- SAM2 propagation runs on the **main thread** — Fusion will appear frozen for 30–120 seconds while it runs. This is expected. Do not click anything.
- If `sam_points` were lost (Resolve restart), propagation gets no dots and returns `{}` — static gate fallback kicks in silently. Check the log for `SAM2 video propagation: N per-frame masks ready`.
- If the TIFF export failed (ExportCurrentFrameAsStill issues), `braw_frames_dir` is `None`, code returns early with error in status. No processing happens.

### 2. Resolve Shutdown Hang — FIXED
**Problem:** When Resolve closed, `on_close` fired → `disp.ExitLoop()` → `disp.RunLoop()` returned → code hit `win.Hide()` at line 2476. With Resolve mid-teardown, `win.Hide()` blocked indefinitely on a dead window. `os._exit(0)` below it never ran.

**Fix 1:** `on_close` now calls `os._exit(0)` directly instead of `disp.ExitLoop()`. Never reaches `win.Hide()`.

**Fix 2:** `_cleanup_on_exit` (atexit handler) now calls `os._exit(0)` at the end — covers the case where Resolve terminates Python without firing the Close event (crash, force-quit).

**What can go wrong:**
- `os._exit(0)` inside an event handler is unusual. If Fusion's event system does something after the handler returns, it won't happen. In practice this is fine — we just need the process to die.
- `_cleanup_on_exit` calling `os._exit(0)` skips all remaining atexit handlers. We only have one, so this is fine.

### 4. Soften Mask Feature (Phase 1)
**What was built:**
- New function `_soften_sam2_mask(mask_float32, soften=0)` — Gaussian blur on the mask after dilation
- `soften=1` → 3×3 kernel, `soften=5` → 11×11 kernel, `soften=10` → 21×21 kernel
- sigma = soften × 0.5
- Applied at all three `_dilate_sam2_mask` call sites (static gate loader, BRAW sync loop, background thread loop)
- `sam2_soften` added to `get_settings()` dict

**What can go wrong:**
- At `soften=0` the function returns immediately (no-op) — safe.
- Very high values (15–20) with small footage will turn the mask into a soft blob. That's user error, not a bug.
- The Gaussian kernel `sz = soften * 2 + 1` is always odd — no OpenCV error possible.

## WHAT CAN GO WRONG — GENERAL

1. **BRAW export via ExportCurrentFrameAsStill fails silently** — Resolve 18.5+ required. If clips are offline or project FPS doesn't match clip FPS, TIFFs may not write. Check log for `CK_BrawFB_` directory and frame count.

2. **SAM2 weights missing** — `run_sam2_video_propagation` imports `sam2.build_sam` and loads weights from `sam2_weights/sam2.1_hiera_small.pt`. If that file is missing, the import crashes and masks fall back to static gate. Log will show the exception.

3. **CUDA OOM during SAM2 propagation on long ranges** — `offload_video_to_cpu=True` is set but VRAM can still run out on very long ranges (200+ frames at 4K). If this happens, SAM2 crashes silently and falls back to static gate.

4. **Slider IDs** — `Sam2SoftenInput`, `Sam2MarginInput`, `RefinerInput` are new widget IDs. If any settings-loading code tries to read old SpinBox IDs and they don't exist, it will throw a KeyError. Currently `get_settings()` reads from `Sam2Margin`, `Sam2Soften`, `RefinerStrength` (the SLIDER IDs, not the SpinBox input IDs) — so this is safe.

### NEXT BUILD: Viewer Scrubber (Option A)
**Goal:** Add a scrubber timeline to the live viewer so the user can drag through all exported TIFF frames and see the mask + Margin + Soften applied live — before running Process Range.

**How it works:**
- After BRAW export, `braw_frames_dir` holds the full TIFF sequence
- Pass this path to the viewer (via session file or launch argument)
- Viewer shows a frame slider (0 → N-1 frames)
- Each scrub position: loads the TIFF, runs `_dilate_sam2_mask` + `_soften_sam2_mask` with current slider values, composites over checkerboard, displays
- Main panel Margin/Soften sliders write to the session file → viewer polls → rerenders current frame live
- No re-export needed — TIFFs are already on disk

**Files to touch:**
- `preview_viewer_v2.py` — add scrubber widget, polling loop for settings, per-frame mask apply
- `CorridorKey.py` — write `braw_frames_dir` path and mask settings to session file after export; pass to viewer on launch

**What can go wrong:**
- TIFFs are in a temp folder that gets deleted after Process Range (`shutil.rmtree` at line ~2144). Need to defer or skip cleanup when viewer is open.
- Viewer is a separate Python process — polling the session file is the right IPC mechanism (already used for live_params.json).
- 4K TIFFs are large (~24MB each). Loading one per scrub tick is fine; don't preload all into RAM.

### Phase 2: Directional Softness
**Goal:** Instead of uniform Gaussian blur, blur the mask in a specific direction (angle 0–360°) to simulate motion blur smear on the mask boundary.

**How it works:**
- Create a line-shaped blur kernel at the given angle using `cv2.filter2D()`
- Add two controls: `Soften Direction` (angle, 0–360°) and `Soften Amount` (strength)
- Add to `_soften_sam2_mask()` as a second blur pass — or replace the Gaussian with directional when angle ≠ 0

**Files to touch:** `_soften_sam2_mask()`, `get_settings()`, UI sliders

### Phase 3: Keyframe System for Soften + Direction
**Goal:** Set Soften/Direction at 3 timeline keyframes (before motion, peak, recovery). Plugin interpolates between them per frame during Process Range.

**How it works:**
- UI: `Set Key` button stores `{frame: N, soften: X, direction: Y}` in a list
- During processing: for each frame, find surrounding keyframes, lerp the values
- Simple list stored in session JSON, editable in a small table in the panel
- No Resolve keyframe API needed — it's all internal to the plugin

**Files to touch:** `CorridorKey.py` — new keyframe list UI, `on_set_keyframe()` handler, interpolation function called per-frame in both processing paths
