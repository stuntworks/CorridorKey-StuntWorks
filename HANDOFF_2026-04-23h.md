## DESPECKLE / SAM2 NOT RENDERING ON FIRST OPEN — FIX IN VIEWER

**Root cause:** `QTimer.singleShot(0, ...)` fires the initial render before the poll (50ms) has loaded `live_params.json`. The viewer renders with defaults, then `on_update` overwrites the params — but the repaint from `on_update` may not show the correct state until the user cycles the checkbox.

**Fix (in `preview_viewer_v2.py`, function `_run_persistent`):**

After `win.show()`, add:
```python
QtCore.QTimer.singleShot(300, win._render_now)
```

This delays the forced initial render until after the poll has loaded `live_params.json` (50ms) and `on_update` has run. The viewer shows correct despeckle state on first open.

Also: remove the `QTimer.singleShot(0, ...)` line inside `_build_slider_panel` (line ~710) that fires `_on_despeckle_toggled` — it's redundant and races with the poll.

## SLIDER RESISTANCE — FIX IN VIEWER

**Root cause:** `_on_despeckle_size_changed` and despill slider handlers call `_render_now()` directly on every `valueChanged` event. On a 4K frame (4096×2160), each render takes 50–300ms. The slider fires 20+ events per drag → UI locks up → feels "stuck."

**Fix (in `preview_viewer_v2.py`):**

In `_on_despeckle_size_changed` and any other slider `_on_*_changed` handler that currently calls `_render_now()`:
Replace `self._render_now()` with `self._schedule_render()`.

`_schedule_render()` uses a 60ms single-shot timer — only renders 60ms after the last slider event, so dragging is smooth.

Note: `_on_despeckle_toggled` (checkbox) should KEEP `_render_now()` — it's a discrete toggle, not a continuous drag, so immediate render is correct there.
